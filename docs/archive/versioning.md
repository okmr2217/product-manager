# バージョン更新作業ガイド（Claude Code 参照用）

新しいバージョンを追加するときに Claude Code が参照する共通ガイド。
新しいバージョンを指示するときは、この方針に従って作業すること。

---

## 基本方針

### 1. バージョン番号の正本

- バージョン番号の正本は **`package.json` の `version`** とする
- バージョン番号を他ファイルで重複管理しない
- アプリ内のバージョン表示箇所では `package.json` の version を参照する（ハードコード禁止）

### 2. 変更履歴の管理体制

変更履歴は **二段階** で管理する。

| 段階 | 場所 | 役割 | 管理者 |
| --- | --- | --- | --- |
| ① 下書き | `CHANGELOG.md`（リポジトリ内） | Claude Code が Git 調査結果を書き出す中間ファイル | Claude Code |
| ② 正本 | プロダクト管理ツールの Release テーブル | ブログにも公開される最終的な変更履歴 | 人間が転記 |

- `CHANGELOG.md` は **転記用の下書き** であり、変更履歴の正本ではない
- 正本はプロダクト管理ツールの Release テーブルに登録されたデータとする
- Claude Code の作業範囲は `package.json` の更新と `CHANGELOG.md` への書き出しまで
- プロダクト管理ツールへの登録は人間が行う

### 3. Git ベースで調査する

- 新しいバージョンを追加するときは、**必ず Git 履歴と差分を確認してから** CHANGELOG を書く
- 推測で変更内容を書かない
- コミットメッセージだけでなく、必要に応じてファイル差分も確認する

---

## バージョン更新時の標準作業手順

新しいバージョンを追加するときは、必ず以下の順序で進めること。

### Claude Code の作業（自動）

1. `package.json` の現在 version を確認する
2. `CHANGELOG.md` の既存内容を確認する
3. 今回の対象範囲となる基準コミットまたは前バージョン時点を特定する
4. 基準点から `HEAD` までのコミット履歴と差分を調査する
5. 変更内容を、ユーザー視点で意味のある粒度に要約する
6. `package.json` の `version` を新しい番号に更新する
7. `CHANGELOG.md` に新しいバージョンを **Release テーブル対応フォーマット** で追記する
8. アプリ内のバージョン表示箇所に反映されることを確認する
9. 必要なら関連する import / utility を調整する
10. 最後にコミットする

### 人間の作業（手動）

11. `CHANGELOG.md` の内容をプロダクト管理ツールのリリースノート作成画面に転記する
12. 必要に応じて文面を調整し、公開する（`isDraft: false` にする）

## CHANGELOG.md のフォーマット

### Release テーブル対応フォーマット

プロダクト管理ツールへの転記を前提に、Release テーブルのフィールドに対応した構造で記述する。

```md
# Changelog

## x.y.z - YYYY-MM-DD

- **type**: MAJOR | MINOR | PATCH | HOTFIX
- **title**: リリースタイトル（Release.title に転記）

### 変更内容

- 変更内容1
- 変更内容2
- 変更内容3
```

各フィールドの対応:

| CHANGELOG の項目 | Release テーブルのフィールド | 説明 |
| --- | --- | --- |
| 見出しのバージョン (`x.y.z`) | `version` | バージョン文字列（例: v1.2.0） |
| 見出しの日付 (`YYYY-MM-DD`) | `releaseDate` | リリース日 |
| `type` | `type` | MAJOR / MINOR / PATCH / HOTFIX |
| `title` | `title` | リリースタイトル |
| 「変更内容」セクション | `content` | マークダウンで記述した変更内容 |

### 記述例

```md
# Changelog

## 1.3.0 - 2025-04-01

- **type**: MINOR
- **title**: タスクフィルタとダークモード対応

### 変更内容

- タスク一覧に優先度フィルタを追加
- ダークモードに対応（システム設定に連動）
- ダッシュボードのカード表示を最適化
- 画像アップロード時のプログレス表示を追加

## 1.2.1 - 2025-03-20

- **type**: PATCH
- **title**: 画像管理の不具合修正

### 変更内容

- サムネイル指定が保存されない不具合を修正
- 画像の並び替えが反映されないケースを修正
```

### type の判断基準

| type | 判断基準 | 例 |
| --- | --- | --- |
| MAJOR | 大きな機能追加、破壊的変更、設計の大幅な見直し | 新しい管理機能の追加、DB スキーマの大幅変更 |
| MINOR | 機能追加、目に見える改善 | 新しいフィルタ、UI の改善、新しい表示項目 |
| PATCH | バグ修正、軽微な調整 | 表示崩れの修正、バリデーションの修正 |
| HOTFIX | 緊急のバグ修正（本番で問題が発生） | データ消失バグ、認証の不具合 |

### 変更内容のまとめ方

#### 記述方針

- 箇条書きは簡潔でよい
- 実装の内部事情ではなく、プロダクト上の変更が分かるようにする
- 同じ意味の変更を細かく分割しすぎず、適度にまとめる
- 以前のバージョンと重複しないようにする

#### まとめ方の観点

以下の観点で整理すると分かりやすい。

- 機能追加
- UI 改善
- 操作性改善
- 表示改善
- 導線改善
- 画面別の改善（ダッシュボード / 詳細 / フォーム等）
- 細かな不具合修正や挙動調整

#### 避けること

- Git のコミットメッセージをそのまま羅列すること
- 技術詳細だけで終わること
- 実際に確認していない内容を書くこと

---

## 実装時の注意

- バージョン管理の正本は `package.json` のみ
- `CHANGELOG.md` は転記用の下書きであり、プロダクト管理ツールへの登録をもって正式な変更履歴となる
- 将来は `package.json` と `CHANGELOG.md` を更新すれば済む状態を保つ
- 既存の UI トーンを崩さない
- 不要な仕組みや自動化を追加しない
- バージョン更新のたびに、必ず Git 調査を先に行う

---

## 作業完了時の報告

Claude Code は作業完了後、以下を簡潔に報告すること。

1. 今回のバージョンに含めた変更内容の要約
2. 更新したファイル一覧
3. アプリ内でバージョンをどこから・どのように参照しているか
4. 作成したコミットメッセージ
5. **プロダクト管理ツールへの転記リマインド**（「CHANGELOG.md の内容をプロダクト管理ツールに登録してください」）

---

## コミット方針

```
chore: バージョンを x.y.z に更新し CHANGELOG を追記
```

## 新しいバージョン作業を依頼するときのテンプレート

以下を埋めて使うこと。

```md
バージョンを **{{NEW_VERSION}}** に更新してください。

この作業では、必ず `docs/archive/versioning.md` を参照して進めてください。

## 今回の前提
- 前バージョン: **{{PREVIOUS_VERSION}}**
- 今回の対象範囲: **{{RANGE_DESCRIPTION}}**
- 基準コミット: **{{BASE_COMMIT_OR_TAG}}**
- 完成日: **{{RELEASE_DATE}}**

## やってほしいこと
1. Git 履歴と差分を確認して、前バージョンから現在までの変更内容を調査
2. 変更内容をユーザー視点で要約
3. `package.json` の `version` を **{{NEW_VERSION}}** に更新
4. `CHANGELOG.md` に **{{NEW_VERSION}}** を Release テーブル対応フォーマットで追記
5. アプリ内のバージョン表示に反映されることを確認
6. 必要なら表示や参照コードを調整
7. 最後にコミット

## CHANGELOG の記載方針
- Release テーブル対応フォーマット（type, title, 変更内容）で書く
- 実際に調査した内容に基づいて書く
- 技術詳細ではなく、プロダクト上の変更が分かる形で簡潔にまとめる
- 前バージョンと重複しないようにする

## 作業完了後に報告してほしいこと
1. {{NEW_VERSION}} に含めた変更内容の要約
2. 更新したファイル一覧
3. アプリ内でどのように version を参照しているか
4. 作成したコミットメッセージ
5. プロダクト管理ツールへの転記リマインド
```

---

## 補足

過去バージョンの整理や、どのコミット範囲をどのバージョンに含めるかは、
必ず今回の依頼文に明記された内容を優先すること。

不明な場合は、まず既存の `CHANGELOG.md` と Git 履歴から整合する範囲を確認し、
推測ではなく事実ベースでまとめること。
